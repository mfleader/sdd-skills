## Code Review Guide (30 minutes)

> This section guides a code reviewer through the implementation changes,
> focusing on high-level questions that need human judgment.

**Changed files:** 4 files changed. Core changes in the backtrace command file (Sections 6, 7, 11), plus data model update, version bump, and SKILL.md regeneration.

### Understanding the changes (8 min)

Start with `.specify/extensions/backtrace/commands/speckit.backtrace.trace.md`, Section 6 (line 124). This is where all the logic lives. The old Section 6 was a flat list of 4 tracing categories. The new Section 6 is a 5-step pipeline: barrier walk, addition type determination, tracing categories, certainty assignment, ProposedAddition output.

Then read Section 7 (line 216) for the one-line auditor skepticism addition, and Section 11 (lines 362-410) for the output format changes and cluster detection.

- The barrier walk (Step 1) and addition type (Step 2) are the most consequential changes. Do the 4 steps of the walk (FR → AC → SC → task) provide enough guidance for an LLM to execute consistently, or are the instructions too abstract?
- Step 2 says "the barrier walk result takes precedence over category-specific defaults." Is the precedence relationship clear enough in the instruction text, or could an LLM misinterpret it and revert to category defaults?

### Key decisions that need your eyes (12 min)

**Barrier walk as semantic association** (Section 6, Step 1, line 132, relates to [FR-018](spec.md#functional-requirements))

Relevance is "determined by semantic association, not keyword matching." This delegates a critical judgment call to the LLM. The alternative was keyword matching (rejected as brittle) or embedding similarity (no infrastructure).
- Question: Is "semantic association" precise enough as an instruction, or would examples of what constitutes a relevant vs irrelevant FR help the LLM execute more consistently?

**Certainty assignment from barrier walk output** (Section 6, Step 4, lines 164-170, relates to [FR-022](spec.md#functional-requirements))

The three certainty levels map to barrier walk outcomes: explicit scope match = direct, domain-related = plausible, inferential = uncertain. The concern from the [improvement research](improvement-research.md#i-002-categorical-trace-certainty) was whether the LLM would collapse most traces into `plausible` as a safe middle ground.
- Question: Should the instructions include a bias toward `direct` (most traces should be direct if the barrier walk works) or let the distribution emerge naturally?

**Cluster warnings placement** (Section 11, lines 400-410, relates to [FR-026](spec.md#functional-requirements))

Cluster warnings appear after Applied Additions but before Reset Trigger Warnings. This ordering means the user sees: what was applied, whether there are systemic patterns, and whether scope changed.
- Question: Is this the right ordering, or should cluster warnings come after rejected additions (so the user sees the full picture before the pattern analysis)?

### Areas where I'm less certain (5 min)

- Section 6, Step 3 (lines 152-162): The tracing categories were reframed from "identify which FR/AC/SC" to "frame the addition as a requirement/criterion/constraint/boundary." This is a subtle but meaningful change. The old phrasing told the LLM *what to find*; the new phrasing tells it *how to frame* the addition. Both are needed, but the "what to find" guidance was lost. An LLM might frame an addition correctly but target the wrong spec item.

- Section 7, line 216: The auditor skepticism instruction says "prefer reject or revise when the causal link is weak." This is subjective. Different LLM invocations may interpret "weak causal link" differently, leading to inconsistent auditor behavior across runs.

- SKILL.md regeneration: The SKILL.md was regenerated by extracting the command file body and prepending the frontmatter. If the specify CLI's `_register_extension_skills()` method has a different regeneration logic (e.g., it wraps content differently), the hand-regenerated SKILL.md may drift from what `specify extension add` would produce.

### Deviations and risks (5 min)

No deviations from [plan.md](plan.md) were identified. The implementation follows the plan's command file change descriptions exactly.

- Risk: The barrier walk adds ~60 lines of instruction to Section 6. The total command file is now ~470 lines. Claude Code skills are executed by reading the full file into context. At what point does instruction length degrade execution quality? The [improvement research](improvement-research.md) noted that F-CoT (input structuring) was rejected as premature, but the command file grew 12% with this change.
