# Data Model: Backtrace Extension

## Entities

### GapFinding (input)

Source: gap-audit `--output` or user-provided JSON file.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| classification | string | yes | `blocking` or `non-blocking` |
| category | string | yes | Gap category (e.g., `Weak ACs`, `Missing error paths`) |
| description | string | yes | What the gap is |
| evidence | string | yes | Quote from spec proving the gap exists |
| suggested_fix | string | yes | Specific improvement suggestion |

### ProposedAddition (internal)

Generated by backtrace during tracing. Sent to auditor for review.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| addition_id | string | yes | Unique ID: finding index + sequential suffix (e.g., "1.1", "1.2", "2.1") |
| finding_ref | string | yes | Reference to the GapFinding this addresses |
| target_artifact | string | yes | `spec.md` or `tasks.md` |
| target_section | string | yes | Section heading where addition goes |
| addition_type | string | yes | One of: `new_FR`, `amended_FR`, `new_AC`, `amended_AC`, `new_SC`, `new_task` |
| content | string | yes | The actual text to add or amend |
| rationale | string | yes | Why this addition addresses the gap |

### AuditorVerdict (output from subagent)

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| addition_ref | string | yes | Matches the `addition_id` of the ProposedAddition being reviewed |
| verdict | string | yes | `approve`, `reject`, or `revise` |
| reason | string | conditional | Required when verdict is `reject`, optional for `approve` and `revise` |
| revision | string | conditional | Required when verdict is `revise` |

| File | Scope | Read | Write |

| `.*-findings.json` (e.g., `.gap-audit-spec-findings.json`) | spec | yes (input) | no |
