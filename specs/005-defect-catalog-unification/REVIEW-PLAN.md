## What This Spec Does

The exploratory test skill's feedback loop is broken: it looks for defect patterns in a per-spec file that nothing produces. Meanwhile, the gap-audit skill reads a separate patterns file with a different entry format. This spec unifies both into a single project-level `specs/defect-catalog.md` with consumer-specific sections and a converged entry format, fixing the broken loop and eliminating the format divergence.

**In scope:** Changing where both skills read catalog patterns from, defining the unified file format contract, updating both extension READMEs, bumping gap-audit to 1.0.0 for the breaking path change.

**Out of scope:** Modifying the wh40k pattern collector (adapts to the contract separately), changing findings output schemas, building a generic collector, per-spec catalog files.

## Bigger Picture

This is the fourth extension spec in the sdd-skills project (after gap-audit, backtrace, and exploratory-test). The pattern catalog system originated in the wh40k project as `sdd-patterns.md`, a single file with sections for different consumers. When the skills were packaged as speckit extensions, the file was split: gap-audit got `gap-patterns.md` with a simplified schema, and the exploratory test skill was pointed at a per-spec path that doesn't exist. This spec reconnects the pieces.

The wh40k collector (`sdd_collect_patterns.py`) will need a follow-up change to produce the converged format. That's tracked in wh40k, not here. The open thread about backtrace not being able to consume exploratory findings (schema incompatibility) is also still unresolved and may warrant its own spec.

## Spec Review Guide (30 minutes)

> Focus your time on the contract definition and breaking change implications. The code changes themselves are straightforward path swaps in markdown command files.

### Understanding the approach (8 min)

Read the [Defect Catalog Contract](spec.md#defect-catalog-contract) section. This is the heart of the spec: the format that both skills read and that external collectors must produce. As you read:

- Does the converged entry format (Category + Trigger + Occurrences + Source + Also known as + Remediation + Example) cover everything both consumers need? The original gap-patterns.md had only Category/Trigger, while sdd-patterns.md had only Occurrences/Source/Remediation/Example. The union adds two fields the collector doesn't currently produce.
- Is TOML frontmatter the right choice? The skills ignore it, so it's purely for external tooling. Is there tooling that would actually use it today?
- The [section table](spec.md#defect-catalog-contract) defines 5 sections but only 2 are consumed by skills. Should the contract specify all 5 or just the 2 that matter?

### Key decisions that need your eyes (12 min)

**Breaking change with clean break** ([FR-009](spec.md#functional-requirements))

Gap-audit switches from `specs/gap-patterns.md` to `specs/defect-catalog.md` with no deprecation period or fallback. Projects with the old file lose pattern support until they migrate. The version bump to 1.0.0 signals this.
- Question: Is a clean break acceptable, or should there be a one-release deprecation period where both paths are tried?

**Required vs optional fields** ([Entry format](spec.md#defect-catalog-contract))

Category and Trigger are required, everything else is optional. But Category and Trigger are the two fields the collector doesn't currently produce. A catalog generated by today's collector would fail gap-audit validation (FR-006).
- Question: Should Category and Trigger be optional during a transition period, with a future spec making them required once the collector catches up?

**Asymmetric validation** ([FR-006](spec.md#functional-requirements) vs [FR-007](spec.md#functional-requirements))

Gap-audit validates structured fields (Category, Trigger). Exploratory test only checks that the section is non-empty. The rationale is that gap-audit does two-part matching while exploratory test uses prose injection.
- Question: Is this asymmetry intentional and justified, or will it confuse maintainers who expect consistent behavior from two skills reading the same file?

### Areas where I'm less certain (5 min)

- [FR-003](spec.md#functional-requirements): The spec says skills extract sections "by H2 heading only" and skip TOML frontmatter. But these are LLM-executed commands (markdown prompt files), not scripts. The "parsing" happens when the LLM reads the command file and follows the instructions. I interpreted this as: the command file instructs the LLM to find the H2 heading and extract content until the next H2. But an LLM might interpret `+++` blocks differently than a parser would. Is the instruction precise enough?

- [Plan Phase 1](plan.md#phase-1-create-unified-catalog-file): The plan says to migrate 3 patterns from gap-patterns.md into the new format, adding Category and Trigger fields. But the current gap-patterns.md already has Category and Trigger. The new fields that need to be added are Occurrences, Source, Also known as, Remediation, and Example. The migration is straightforward but the plan description could be clearer about which direction the field additions go.

### Risks and open questions (5 min)

- The wh40k collector doesn't produce Category or Trigger today. If it's not updated, the unified catalog will fail gap-audit's validation ([FR-006](spec.md#functional-requirements)). How tightly should the sdd-skills release be coordinated with the collector update?
- The spec defines "project root" as the working directory ([Assumptions](spec.md#assumptions)). If a skill is invoked from a subdirectory or a git worktree with a different root, `specs/defect-catalog.md` won't resolve. Is this a real risk or a theoretical one?
- After this change, `specs/gap-patterns.md` becomes dead weight in any project that has it. Should the exploratory-test or gap-audit README include a migration guide, or is the breaking change + version bump sufficient signal?
